QUAMS

Site éphémère pour un événement gaming en région parisienne

Description du Projet

QUAMS est un site éphémère développé lors de mon stage en première année pour un événement gaming en région parisienne. Le projet met en œuvre des technologies modernes pour offrir une expérience utilisateur fluide et sécurisée.

Fonctionnalités

Inscription et Authentification

Système complet d'inscription et de connexion pour les participants à l'événement.

Paiement en Ligne

Intégration de l'API Stripe pour gérer les paiements des billets d'entrée de manière sécurisée.

Gestion des Événements

Interface d'administration pour gérer les tournois, les participants et les résultats.

Technologies Utilisées

PHP

Développement backend avec PHP pour la logique métier et la gestion des données

MySQL

Base de données relationnelle pour stocker les informations des utilisateurs et des événements

API Stripe

Intégration de l'API de paiement Stripe pour les transactions sécurisées

Interface d'intégration de l'API Stripe pour la gestion sécurisée des paiements des billets d'entrée aux tournois

Modèle Conceptuel de Données

Entités Principales

UTILISATEUR

Entité centrale qui représente tous les acteurs du système :

  • Joueurs
  • Spectateurs
  • Administrateurs

L'attribut type_utilisateur permet de différencier les rôles

ÉVÉNEMENT

Représente l'événement global qui peut contenir plusieurs tournois :

  • Gestion des dates
  • Contrôle des inscriptions
  • Organisation des tournois

Permet de gérer l'ouverture/fermeture des inscriptions

TOURNOI

Compétition organisée dans le cadre d'un événement :

  • Lié à un événement spécifique
  • Associé à un jeu particulier
  • Gestion des formats de compétition

Relations :

  • 1 tournoi → 1 événement
  • 1 tournoi → 1 jeu
  • 1 tournoi → N participants
  • 1 tournoi → N matchs

ÉQUIPE

Entité séparée pour gérer les équipes multi-joueurs :

  • Nom de l'équipe
  • Logo
  • Statut

Reliée aux utilisateurs via la table COMPOSITION

Relations Importantes

Participation aux Tournois

PARTICIPANT : Table de liaison entre UTILISATEUR, TOURNOI et optionnellement ÉQUIPE

Permet de gérer les inscriptions solo ou en équipe

Composition des Équipes

COMPOSITION : Table de liaison N:N entre UTILISATEUR et ÉQUIPE

  • Rôle dans l'équipe (capitaine/membre)
  • Date d'intégration

Gestion des Matchs

MATCH : Peut opposer soit des joueurs individuels, soit des équipes

Les FK id_joueur_1/2 et id_equipe_1/2 sont nullables selon le type de tournoi

Billetterie et Paiements

BILLET : Lié à un utilisateur et un événement

PAIEMENT : Relation 1:1 avec BILLET pour tracer chaque transaction

Streaming

STREAM : Optionnel, lié aux matchs pour la diffusion en direct

Cardinalités Clés

  • 1 Utilisateur → N Billets (un utilisateur peut acheter plusieurs billets)
  • 1 Événement → N Tournois (un événement peut avoir plusieurs tournois)
  • 1 Tournoi → N Matchs (un tournoi contient plusieurs matchs)
  • N Utilisateurs ↔ N Équipes (relation many-to-many via COMPOSITION)
  • 1 Billet → 1 Paiement (chaque billet a un paiement unique)

Points d'Attention pour l'Implémentation

  • Gestion des tournois solo vs équipe : Les FK dans MATCH sont conditionnelles selon le type de tournoi
  • Intégrité référentielle : Prévoir des contraintes pour éviter qu'un joueur soit inscrit plusieurs fois au même tournoi
  • Gestion des états : Les ENUM permettent de contrôler les flux (événements ouverts/fermés, billets valides/utilisés, etc.)
  • Scalabilité : La structure permet d'ajouter facilement de nouveaux types de billets, méthodes de paiement, etc.

Ce modèle couvre tous les besoins fonctionnels tout en restant flexible pour les évolutions futures !
